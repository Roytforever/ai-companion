# 知识库方案决策

## 问题

对于女娲蒸馏出来的人物，需要存储什么才能让"后续持续蒸馏"达到最大效果？

答案是：**蒸馏全生命周期中产生的所有结构化素材**。

## 存储方案对比

### 方案A：本地 JSON 文件系统（✅ 推荐）

```
knowledge_base/data/
├── index.json              # 全局索引
├── elon-musk_20260615_143022.json  # 素材记录
├── elon-musk_conversation_1.json    # 对话精华
└── ...
```

**优点**：
- ✅ **零外部依赖**，开箱即用，纯文件系统
- ✅ 每个文件可独立读写，便于人工审查
- ✅ 项目clone下来就能用，适合练手项目
- ✅ 可与 Git 版本控制配合（追溯蒸馏过程）

**缺点**：
- ❌ 大量数据后搜索性能下降
- ❌ 没有结构化查询能力（如"找所有标签包含'投资'的人物"）

### 方案B：SQLite 数据库

**优点**：
- 结构化查询（SQL）
- 单文件存储，便于迁移
- 轻量级，无服务端

**缺点**：
- 人工查看不便（需要工具打开）
- 不适合存储大文本（BLOB处理麻烦）

### 方案C：向量数据库（Chroma/Qdrant）

**优点**：
- 语义搜索，找"思维方式相似"的人物
- 适合 RAG 增强蒸馏效果

**缺点**：
- ❌ 外部依赖大，不适合练手项目
- ❌ 需要 Embedding 模型

## 结论

**选择方案A：本地 JSON 文件系统**

原因：
1. **练手项目的核心定位** — 不引入任何外部依赖
2. **蒸馏数据天然适合文件存储** — 每个person的素材是独立、自包含的
3. **渐进式升级** — 未来想升级时，从JSON到SQLite/向量库的迁移路径清晰

## 存储什么

对每个人物，存储三类数据：

| 类型 | 存储内容 | 用途 |
|------|---------|------|
| **原始素材** | 用户提供的文章/链接/笔记 | 蒸馏的基础原料 |
| **对话精华** | 用户与此人物模式的聊天记录 | 持续优化角色表现 |
| **蒸馏快照** | 各版本SKILL.md的快照 | 版本回溯、增量更新 |

## 持续蒸馏流程

```
用户提供素材 → 保存到知识库 → 女娲读取素材 → 生成初始SKILL
        ↑                                              |
        └──── 用户与此人物对话 ──→ 提炼精华 ──→ 更新SKILL
```
